# 答辩常见问题

## 1. 为什么使用 WebView + H5 Canvas？

本项目最终交付是 Android APK，WebView 负责把本地 H5 页面打包进手机应用，Canvas 负责读取像素和绘制直方图。这样 Android 层只处理安装、启动、本地文件选择等壳能力，算法逻辑集中在 H5 层，边界比较清楚。对课程项目来说，这条路线实现成本可控，也便于用脚本验证算法结果。整个应用离线运行，不依赖后端页面。

## 2. 为什么是 256 bins？

灰度图像通常把亮度表示为 `0-255` 的整数范围，一共 256 个可能值。每个 bin 对应一个灰度值，统计它在图片中出现了多少次。这样输出和课程要求一致，也方便检查 bins 总和是否等于图片像素总数。少于 256 个 bins 会丢失灰度精度。

## 3. 为什么归一化到 0-100？

课程要求输出 `256x100` 黑白直方图，所以纵向高度需要落在 `0-100` 范围。程序先找到当前图片中计数最大的 bin，再把所有 bin 按这个最大值缩放。归一化只影响绘图高度，不改变原始像素计数。这样不同尺寸图片都能画进固定高度的结果图。

## 4. 为什么使用 gray = 0.299R + 0.587G + 0.114B？

这是课程任务书指定的灰度公式，项目必须按这个公式实现。它给绿色通道更高权重，红色次之，蓝色较低，符合常见亮度感知计算方式。代码和测试都围绕这个公式固定，不使用平均值或其他库默认公式。这样答辩时公式、代码和测试口径一致。

## 5. 计时范围包含哪些步骤？

计时范围覆盖直方图生成的完整链路，包括把图片绘制到处理 Canvas、读取 RGBA 像素、计算灰度、统计 256 bins、归一化到 `0-100`，以及绘制 `256x100` 直方图。不只统计最后一次 draw call。这样耗时更接近用户点击选图后等待结果的真实成本。页面展示的目标是说明直方图生成耗时应控制在 300ms 以内。

## 6. 为什么低效对照版本更慢？

低效对照版本用于说明普通数组、对象分配、重复灰度计算和重复扫描会增加处理成本。它不是历史生产版本，也不是为了冒充真实早期代码。正式交付版本采用 typed array、单次遍历和固定 `256x100` 渲染，减少不必要的对象和循环。两者仍保持相同公式、bins、归一化和输出尺寸，保证对比公平。

## 7. 如何证明当前优化版本结果没有错？

项目用自动化测试验证灰度公式、256 bins、像素总数、`0-100` 归一化和 `256x100` 绘制规则。`npm run test:baseline` 会检查低效对照版本和当前优化版本输出是否一致。也就是说，性能对比不是牺牲结果换速度。答辩时可以说明“先保证结果一致，再比较耗时”。

## 8. 为什么不使用 OpenCV / AI / 后端？

课程题目要求的是移动端本地图像直方图计算，不需要 AI 推理、OpenCV 算子库或后端服务。引入这些概念会增加部署和解释成本，也可能让离线 APK 验收变复杂。本项目用 WebView + H5 Canvas 就能完成选图、像素读取、统计和绘制。保持方案简单，更利于证明公式、尺寸和耗时边界。

## 9. 如果图片很大超过 300ms 怎么办？

如果图片尺寸很大，像素读取和逐像素统计耗时会随像素数增加，超过 300ms 是需要如实记录的情况。最终报告应写明设备型号、Android 版本、图片尺寸、像素数和实测耗时，不能只挑选好看的数据。演示时建议准备中等尺寸图片保证流程稳定，大图可以作为限制说明。后续优化方向是继续减少遍历和对象开销，而不是改公式或少算像素。
