# 移动端图像直方图分析系统关键技术调研报告

## 1. 文档说明

本文档从需求分析师视角整理项目关键技术调研结果，说明为什么选择 Android WebView + H5 Canvas、为什么保持本地离线处理、为什么使用当前灰度统计与性能优化方案。

调研结论服务于课程交付，不追求覆盖所有图像处理技术。本文只围绕本项目真正使用到的关键技术展开。

## 2. 调研目标

| 目标 | 说明 |
| --- | --- |
| 确定移动端交付形态 | 选择能稳定交付 APK、便于验收的技术路线 |
| 确定图像处理方案 | 满足本地选图、像素读取、灰度统计、直方图绘制 |
| 确定离线运行方案 | 避免服务器、数据库和网络依赖 |
| 确定性能优化方向 | 能解释直方图生成耗时应控制在 300ms 以内，以及低效对照与当前优化差异 |
| 确定测试验证方式 | 让公式、bin 数、归一化和绘制尺寸可验证 |

## 3. 技术路线对比

### 3.1 移动端实现路线

| 方案 | 优点 | 缺点 | 是否采用 |
| --- | --- | --- | --- |
| 原生 Android 全量实现 | 性能好，系统能力完整 | 图像处理和 UI 都要写原生代码，短周期成本较高 | 未采用 |
| Android WebView + H5 Canvas | APK 形态真实，H5 处理图像方便，代码可读，便于快速验证 | WebView 性能受设备影响，需要控制图片和算法实现 | 采用 |
| 纯 Web 页面 | 开发快，调试方便 | 不是 APK，离线移动端交付不够贴合任务 | 未采用 |
| 小程序 | 移动端体验好 | 依赖平台账号、审核和运行环境，不适合当前离线 APK 要求 | 未采用 |

采用 Android WebView + H5 Canvas 的原因：

- 能交付 Android APK；
- 能用 H5 Canvas 直接读取像素和绘制直方图；
- 不需要后端；
- 便于在浏览器预览和脚本测试中验证算法；
- 适合五天周期内完成课程项目闭环。

### 3.2 图像处理技术

| 技术 | 作用 | 项目取舍 |
| --- | --- | --- |
| Canvas `drawImage` | 将图片绘制到处理画布 | 采用 |
| Canvas `getImageData` | 获取 RGBA 像素数组 | 采用 |
| SVG 图表 | 绘制矢量图形 | 不作为核心结果，避免与真实像素统计脱节 |
| 第三方图表库 | 快速生成图表 | 不采用，课程要求固定 `256x100` 黑白输出 |
| WebGL | 更适合并行图形计算 | 当前算法简单，使用成本高，不采用 |

Canvas 是本项目最合适的技术点：它既能读取真实像素，也能按要求输出固定尺寸结果。

## 4. Android WebView 调研

### 4.1 使用价值

Android WebView 允许原生 APK 加载本地 H5 页面。对本项目来说，它承担三件事：

| 能力 | 说明 |
| --- | --- |
| APK 壳 | 让项目以 Android 应用形式安装和启动 |
| 本地资源加载 | 加载 `file:///android_asset/index.html` |
| 系统文件选择 | 通过 WebView 文件选择回调接入手机本地图片 |

### 4.2 风险与处理

| 风险 | 处理方式 |
| --- | --- |
| 不同 Android 版本 WebView 行为略有差异 | 保持功能简单，只依赖基础文件选择和 Canvas 能力 |
| 本地图片权限复杂 | 优先使用系统文件选择器，减少主动申请存储权限 |
| 页面依赖远程资源会导致断网失败 | 所有核心 H5、CSS、JS 打包进 APK assets |
| 大图处理可能卡顿 | 使用 typed array 和单次遍历，演示时准备合适尺寸图片 |

## 5. H5 Canvas 调研

### 5.1 Canvas 适配本项目的原因

Canvas 提供两类关键能力：

1. 把用户选择的图片绘制到隐藏画布；
2. 读取 RGBA 像素数据并绘制结果图。

这两类能力正好对应课程要求中的“统计彩色图像像素”和“生成黑白直方图”。

### 5.2 Canvas 处理链路

```text
图片文件
-> HTMLImageElement
-> drawImage 到 sourceCanvas
-> getImageData 获取 RGBA
-> 遍历像素计算灰度
-> draw 到 histogramCanvas
```

### 5.3 关键注意点

| 注意点 | 说明 |
| --- | --- |
| 源图 Canvas 尺寸 | 应按图片实际尺寸读取像素 |
| 输出 Canvas 尺寸 | 必须固定为 `256x100` |
| 页面展示尺寸 | 可以响应式放大，但不能改变源数据语义 |
| 性能计时 | 应包含 Canvas 像素读取和结果绘制 |

## 6. 灰度算法调研

课程指定公式：

```text
gray = red * 0.299 + green * 0.587 + blue * 0.114
```

这个公式给绿色通道更高权重，符合人眼对绿色更敏感的常见亮度计算方式。项目不能为了视觉偏好替换公式，因为验收关注的是题目指定规则。

项目中的取整策略为 `Math.round`，并通过测试保持一致。统计规则如下：

```text
grayBin = Math.round(gray)
bins[grayBin] += 1
```

约束：

- bin 范围为 `0-255`；
- bin 数量固定为 256；
- 每个像素参与一次统计；
- bins 总和应等于图片像素数。

## 7. 归一化与绘制调研

原始 bin 计数可能很大，不能直接绘制到 100 像素高的结果图上，因此需要归一化。

归一化规则：

```text
maxCount = max(bins)
normalized[i] = bins[i] / maxCount * 100
```

绘制规则：

| 项目 | 规则 |
| --- | --- |
| 宽度 | 256 |
| 高度 | 100 |
| 背景 | 白色 |
| 柱体 | 黑色 |
| 横轴 | 灰度值 `0-255` |
| 纵轴 | 归一化后的相对高度 |

这个规则直接对应课程要求，避免使用普通图表库时出现坐标轴、间距、颜色或尺寸不符合要求的问题。

## 8. 性能优化调研

### 8.1 性能瓶颈来源

图像直方图计算的主要耗时来自：

- Canvas 像素读取；
- 遍历 RGBA 数组；
- 灰度计算；
- bin 统计；
- 归一化；
- 结果绘制。

如果实现中频繁创建对象、重复计算灰度、重复扫描数组，耗时会明显上升。

### 8.2 当前优化策略

| 策略 | 作用 |
| --- | --- |
| `Uint32Array(256)` | 降低统计结构开销 |
| 单次遍历 | 避免重复扫描像素 |
| 一次归一化 | 避免多轮计算最大值和高度 |
| 固定 `256x100` 渲染 | 控制输出规模 |
| 同边界计时 | 保证低效对照版本与当前优化版本对比公平 |

### 8.3 对照实验

项目构造了 `comparison-baseline` 和 `current-optimized` 两种模式：

| 模式 | 说明 |
| --- | --- |
| comparison-baseline | 低效压力对照版本，用普通数组、对象分配、重复计算和重复扫描放大性能问题 |
| current-optimized | 当前优化版本，使用 typed array、单次遍历和固定尺寸渲染 |

已有开发机 benchmark 显示，当前实现策略在三组合成图片上均低于 `300ms`。但这个数据是 Node.js 环境下的同边界策略对照，不替代 Android 真机测试。最终报告建议补充手机实测数据。

## 9. 离线运行调研

课程验收更关注本地 APK 是否可运行，而不是远程服务能力。因此项目采用离线方案：

| 项目 | 处理方式 |
| --- | --- |
| 页面资源 | 打包进 `app/src/main/assets/` |
| 图片来源 | 用户本地选择 |
| 图像处理 | H5 Canvas 本地完成 |
| 数据保存 | 默认不上传；拼接图保存到本机图库 |
| 网络依赖 | 核心功能无网络依赖 |

这种方案也更符合隐私要求：用户图片不离开手机。

## 10. 测试验证调研

为了避免“看起来有图，但结果不一定正确”，项目把关键规则做成脚本检查：

| 验证项 | 方式 |
| --- | --- |
| 灰度公式 | fixture 检查 |
| bin 数量 | 检查数组长度为 256 |
| 像素总数 | 检查 bins 求和等于像素数 |
| 归一化 | 检查高度范围和比例 |
| 绘制尺寸 | 检查输出为 `256x100` |
| 离线资源 | 检查 assets 中无远程依赖 |
| 对照一致性 | 检查低效对照版本与当前优化版本输出一致 |

常用命令：

```bash
npm run test:histogram
npm run test:offline
npm run test:baseline
npm run benchmark:histogram
npm run harness:verify
```

## 11. 技术选型结论

Android WebView + H5 Canvas 是本项目比较稳的选择。它没有原生全量开发那么重，也比纯 Web 更符合 APK 交付要求；Canvas 能直接完成像素读取和固定尺寸绘制，适合课程题目中的直方图计算。

性能方面，当前优化点集中在数据结构和遍历方式上，能解释为什么当前优化版本比低效对照版本更快。后续如果还要增强，优先补真机多尺寸耗时表，而不是继续引入 Web Worker、WebGL 或后端服务。

## 12. 后续建议

- 补充一组或两组 Android 真机耗时数据；
- 在测试报告中明确 benchmark 与真机测试的区别；
- 答辩时优先讲 Canvas 像素读取、灰度公式、256 bin、`256x100` 输出和性能对照；
- 不再引入后端、数据库或云上传，避免偏离课程主线。
