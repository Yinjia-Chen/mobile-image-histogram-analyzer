# 研发自查清单

## 1. 文档说明

本文档用于研发人员在提交代码、打包 APK 或配合答辩前做快速自查。它把需求、测试和项目管理文档中的关键约束整理成可执行检查项。

## 2. 范围自查

| 检查项 | 通过标准 |
| --- | --- |
| 是否仍是 Android APK 交付 | 最终演示以 APK 为准，不只演示浏览器页面 |
| 是否保持本地离线处理 | 不引入后端、数据库、登录、云上传 |
| 是否使用本地图片 | 不依赖网络图片或远程接口 |
| 是否保持单图主流程 | 多图批处理不作为当前核心范围 |
| 是否区分增强功能和核心功能 | 放大、保存、对比是增强，不替代主流程 |

## 3. 算法自查

| 检查项 | 通过标准 |
| --- | --- |
| 灰度公式 | `gray = red * 0.299 + green * 0.587 + blue * 0.114` |
| bin 数量 | 固定 256 |
| 统计完整性 | bins 求和等于像素总数 |
| 取整口径 | 与测试保持一致，当前为 `Math.round` |
| 归一化 | 高度范围 `0-100` |
| 绘制尺寸 | 源 Canvas 为 `256x100` |
| 绘制来源 | 来自真实统计数据，不是装饰图 |

## 4. 性能自查

| 检查项 | 通过标准 |
| --- | --- |
| 计时边界 | 覆盖像素读取、灰度计算、统计、归一化、绘制 |
| 优化方式 | 不改变公式，不跳过像素 |
| baseline 对照 | 同边界、同输入、同输出规则 |
| benchmark 表述 | 明确开发机数据不替代真机 |
| 真机数据 | 测试人员回填后再写成最终结论 |

## 5. Android 自查

| 检查项 | 通过标准 |
| --- | --- |
| WebView 入口 | flavor 入口正确 |
| 文件选择 | `onShowFileChooser` 可回调 Uri 或取消 |
| 本地 assets | HTML、CSS、JS 已打包进 APK |
| 网络权限 | 核心功能不依赖 INTERNET |
| 启动背景 | 深色背景，避免明显白屏 |
| 可并装 | baseline 和 optimized 包名不同 |
| 保存桥 | 只用于用户确认后的拼接图保存 |

## 6. H5 自查

| 检查项 | 通过标准 |
| --- | --- |
| 图片选择入口 | `type="file"` 且 `accept="image/*"` |
| 预览 | 选图后可确认当前图片 |
| 输出页 | 可显示直方图 |
| 指标页 | 可解释 bins、像素数、耗时、性能余量 |
| 协议页 | 可展示公式和规则 |
| 日志页 | 可展示处理过程 |
| 异常处理 | 取消、非图片、读取失败不崩溃 |

## 7. 验证命令自查

提交前建议执行：

```bash
npm run test:histogram
npm run test:offline
npm run test:baseline
npm run benchmark:histogram
npm run check:source-comments
npm run harness:verify
```

涉及 APK 时再执行：

```bash
JAVA_HOME="/Applications/Android Studio.app/Contents/jbr/Contents/Home" \
ANDROID_HOME="/Users/framia/Library/Android/sdk" \
./gradlew --offline assembleBaselineDebug assembleOptimizedDebug
```

## 8. 文档同步自查

| 改动类型 | 需要同步 |
| --- | --- |
| 改算法 | 测试脚本、技术设计、测试报告 |
| 改 APK 入口 | 构建说明、APK 交付记录、README |
| 重打包 APK | APK 大小、SHA256、包名、versionName |
| 改 UI 展示 | README 截图、使用说明、答辩材料 |
| 改性能逻辑 | benchmark、性能说明、测试报告 |
| 完成文档任务 | `docs/研发/tasks-list.md` |

## 9. 答辩自查

| 检查项 | 通过标准 |
| --- | --- |
| 演示图片 | 至少准备普通图、亮/暗图、较大图 |
| 主演示 APK | 优先使用 optimized APK |
| 离线演示 | 现场可断网打开 |
| 讲解主线 | 选图 -> 公式 -> bins -> 归一化 -> `256x100` -> 耗时 |
| 备用方案 | 保留 APK 文件和截图 |

## 10. 自查结论

研发提交的目标不是把功能越堆越多，而是保证核心路径稳定、公式正确、证据完整、演示顺畅。只要改动会影响 APK、算法、性能或文档口径，就必须同步验证和更新记录。
