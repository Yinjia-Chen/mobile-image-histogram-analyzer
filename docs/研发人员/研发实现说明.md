# 移动端图像直方图分析系统研发实现说明

## 1. 文档说明

本文档从研发人员视角说明当前系统已经实现的功能、代码结构、模块边界和实现证据。它承接需求分析、设计、测试和项目管理文档，用于说明“代码里到底做了什么”。

当前项目主形态是 Android WebView APK + H5 Canvas。Android 层负责移动端壳能力，H5 层负责图像处理和展示。项目不包含后端、数据库、登录或云端图片上传。

## 4. Android 层实现

### 4.1 WebView 初始化

`MainActivity` 通过代码创建全屏 WebView，启用 JavaScript、DOM Storage、本地文件访问和 content Uri 访问。入口 URL 来自 `BuildConfig.LOCAL_ENTRY_URL`：

| Flavor | 入口 |
| --- | --- |
| baseline | `file:///android_asset/index-baseline.html` |
| optimized | `file:///android_asset/index.html` |

这样可以在不改 Java 代码的情况下，通过 Gradle flavor 输出两份不同入口的 APK。

### 4.2 本地图片选择

H5 中的 `input[type=file]` 会触发 `WebChromeClient.onShowFileChooser`。Android 层打开系统 `ACTION_GET_CONTENT` 选择器，并将用户选中的图片 Uri 回传给 WebView。

处理原则：

- 每次只选择一张图片；
- 取消选择时回传 `null`，避免 WebView callback 卡住；
- 使用系统选择器，不主动引入复杂存储权限；
- 不从网络读取图片。

### 4.3 保存桥

长按保存拼接图时，H5 在用户确认后把 PNG base64 传给 `AndroidHistogramBridge.saveCompositeImage`。Java 层通过 MediaStore 写入系统图片库。

保存功能是展示增强，不影响核心验收路径。如果某些手机系统限制保存，仍可通过截图完成答辩证据。

## 5. H5 层实现

### 5.1 页面组织

优化版页面按答辩讲解拆成六个视图：

| 视图 | 作用 |
| --- | --- |
| 接入 | 选图、预览、处理状态 |
| 链路 | 展示 Android Shell / H5 Canvas / Algorithm Core / Result Layer |
| 输出 | 展示直方图、Histogram Compare 和结果操作 |
| 指标 | 展示耗时、像素总数、bins、最大计数、性能余量 |
| 协议 | 展示公式、通道、归一化和输出规则 |
| 日志 | 展示处理步骤和状态 |

### 5.2 主流程

```text
用户选择图片
-> 校验文件类型
-> 加载图片对象
-> 显示原图预览
-> 绘制到 sourceCanvas
-> getImageData 读取 RGBA
-> 生成 bins 和 normalized
-> 绘制 histogramCanvas
-> 更新耗时、指标、日志
```

### 5.3 结果对象

研发侧维护的结果对象主要包含：

| 字段 | 说明 |
| --- | --- |
| `bins` | 256 个灰度计数 |
| `normalized` | 归一化到 `0-100` 的高度 |
| `elapsedMs` | 处理耗时 |
| `width` / `height` | 原图尺寸 |
| `pixelCount` | 像素总数 |
| `maxCount` | 最大 bin 计数 |

这些数据既用于页面展示，也用于测试报告和答辩说明。

## 6. Baseline 与 Optimized

| 项 | baseline | optimized |
| --- | --- | --- |
| 定位 | 低效压力对照 | 正式演示版本 |
| 入口 | `index-baseline.html` | `index.html` |
| 统计结构 | 普通数组、对象分配等低效路径 | `Uint32Array(256)` |
| 遍历方式 | 存在重复计算和重复扫描 | 单次遍历为主 |
| 用途 | 说明优化价值 | 课程交付和答辩演示 |

两者必须保持相同公式、相同 256 bins、相同归一化规则和相同 `256x100` 输出语义。

## 8. 实现结论

当前研发实现已经覆盖课程题目的主要求：Android APK、本地选图、灰度公式、256 bin、`0-100` 归一化、`256x100` 黑白直方图、耗时展示和性能对照。

后续研发侧不建议继续加大功能范围。若需要再动代码，应优先修复验收测试发现的问题，并同步更新 APK 产物、哈希、测试记录和任务台账。
