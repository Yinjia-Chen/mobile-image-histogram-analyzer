# 核心算法与性能优化说明

## 1. 文档说明

本文档面向研发和答辩说明，整理项目核心算法、绘制规则、计时边界和性能优化策略。所有说明都以当前实现和测试证据为准，不把开发机 benchmark 写成 Android 真机实测。

## 2. 算法输入输出

| 项 | 内容 |
| --- | --- |
| 输入 | 用户本地选择的一张图片 |
| 中间输入 | Canvas `getImageData` 读取的 RGBA 像素数组 |
| 中间输出 | 256 个灰度 bin、`0-100` 归一化高度 |
| 最终输出 | `256x100` 黑白直方图、生成耗时、指标说明 |

## 3. 灰度计算

课程指定公式为：

```text
gray = red * 0.299 + green * 0.587 + blue * 0.114
```

实现中使用 `Math.round(gray)` 映射到整数 bin，并限制在 `0-255` 范围内。

示意流程：

```text
读取 red、green、blue
-> 计算 gray
-> Math.round(gray)
-> 写入 bins[grayBin]
```

不能替换为其他公式，也不能跳过部分像素来制造更快的结果。

## 4. 256 bin 统计

优化版使用：

```js
const bins = new Uint32Array(256);
```

每个灰度值对应一个 bin：

| bin 索引 | 含义 |
| --- | --- |
| `0` | 黑色或接近黑色的像素 |
| `1-254` | 中间灰度像素 |
| `255` | 白色或接近白色的像素 |

正确性约束：

- `bins.length === 256`；
- 所有像素都参与统计；
- `sum(bins) === image.width * image.height`；
- 每个像素只统计一次。

## 5. 归一化

原始计数不能直接画进 100 像素高的图，所以需要按最大 bin 计数缩放：

```text
maxCount = max(bins)
normalized[i] = bins[i] / maxCount * 100
```

归一化后的高度用于绘制，不改变原始计数。指标页中同时保留像素总数、最大计数和 bins 信息，便于解释结果真实性。

## 6. 绘制规则

直方图源尺寸固定为：

```text
width = 256
height = 100
```

绘制规则：

- 背景为白色；
- 柱体为黑色；
- 横向每一列对应一个灰度值；
- 柱体从底部向上绘制；
- 页面展示可以放大，但源 Canvas 语义不变。

这点很重要：不能为了美观改成普通柱状图，也不能加坐标轴导致输出不再是 `256x100` 黑白直方图。

## 7. 计时边界

页面展示的耗时覆盖完整生成链路：

```text
Canvas 绘制原图
-> getImageData 读取像素
-> 灰度计算
-> 256 bin 统计
-> 0-100 归一化
-> 256x100 直方图绘制
```

不只统计最后的 draw call。这样耗时数据虽然可能更大，但更符合课程对“直方图生成速度”的理解。

## 8. 性能瓶颈

主要耗时来源：

| 环节 | 说明 |
| --- | --- |
| 图片解码 | 浏览器/WebView 将本地文件解码为图片对象 |
| Canvas 绘制 | 将图片画到 sourceCanvas |
| 像素读取 | `getImageData` 从 Canvas 取出 RGBA |
| 像素遍历 | 每 4 个通道处理一个像素 |
| 统计结构 | 数据结构选择影响频繁写入成本 |
| 结果绘制 | 生成 `256x100` 输出 |

## 9. 优化策略

| 策略 | 说明 | 效果 |
| --- | --- | --- |
| TypedArray | 使用 `Uint32Array(256)` 存储 bins | 降低统计结构开销 |
| 单次遍历 | 遍历 RGBA 时直接完成灰度和计数 | 避免重复扫描 |
| 固定输出尺寸 | 输出始终 `256x100` | 避免绘制成本随原图放大 |
| 减少对象分配 | 不为每个像素创建临时对象 | 降低 GC 压力 |
| 同边界对照 | baseline 和 optimized 使用同一计时边界 | 保证性能说明可信 |

## 10. Baseline 对照

baseline 不是正式产品版本，而是故意构造的低效对照，用于说明实现策略差异。

| 对照项 | baseline | optimized |
| --- | --- | --- |
| 统计结构 | 普通数组、对象映射 | `Uint32Array(256)` |
| 计算方式 | 重复灰度计算、重复字符串转换 | 单次遍历计算 |
| 最大值处理 | 重复扫描 | 一次获取 |
| 输出要求 | 与 optimized 一致 | 正式输出 |

如果两者输出不一致，性能对照无效。因此 `npm run test:baseline` 会检查 bins、归一化和渲染缓冲区一致性。

当前开发机 benchmark 记录：

- 真机最终耗时应由测试人员补充。

## 12. 研发注意事项

- 不改灰度公式；
- 不减少参与统计的像素；
- 不把耗时边界缩小到只剩绘制；
- 不把 baseline 当正式版本；
- 不新增远程依赖；
- 改动算法后必须跑 `npm run test:histogram` 和 `npm run test:baseline`。

## 13. 结论

当前核心算法简单但约束明确。研发实现的关键不在于引入复杂库，而在于保持公式、bin 数、归一化和绘制尺寸的稳定，同时用 typed array 和单次遍历降低处理成本。现有实现已经具备答辩说明和测试验证基础。
