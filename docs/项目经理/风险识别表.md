# 移动端图像直方图分析系统风险识别表

## 1. 文档说明

本文档从项目经理视角识别“移动端图像直方图分析系统”在开发、测试、文档和答辩阶段可能遇到的风险，并给出当前状态、预防措施和应急方案。

项目实际只用了五天完成主要内容，因此风险管理采用短周期项目的写法：早期重点盯范围、算法和 APK 主流程，后期重点盯证据、文档一致性和答辩演示稳定性。

## 2. 风险等级口径

| 维度 | 低 | 中 | 高 |
| --- | --- | --- | --- |
| 发生概率 | 偶发，已有控制措施 | 有可能发生，需要跟踪 | 很容易发生或已经出现苗头 |
| 影响程度 | 影响局部体验 | 影响验收说明或部分演示 | 影响核心验收或最终提交 |

综合等级按“概率 + 影响”判断：

| 综合等级 | 处理要求 |
| --- | --- |
| 高 | 必须有明确负责人、预防措施和应急方案 |
| 中 | 需要跟踪，在收尾检查中确认 |
| 低 | 保留记录，避免后续反复踩坑 |

## 3. 五天周期风险重点

| 时间点 | 主要风险 | 管控重点 |
| --- | --- | --- |
| 第 1 天 | 选题跑偏、范围过大 | 明确只做移动端离线直方图，不做后端和登录 |
| 第 2 天 | 架构边界不清、WebView 本地加载不稳定 | Android 只做壳层，H5 负责算法，assets 本地打包 |
| 第 3 天 | 算法结果不准、直方图不是 256 bin | 用固定公式、固定 bin 数和 fixture 测试兜底 |
| 第 4 天 | APK 打包、离线运行和真机验证证据不足 | 输出 APK 记录、离线检查和待回填清单 |
| 第 5 天 | 性能口径、文档口径、答辩口径不一致 | 固化低效对照版本/当前优化版本对照，集中补齐项目管理文档 |

## 6. 当前重点风险

### 6.1 真机回填风险

当前 APK 和自动化测试证据比较完整，但 `stage4-apk-handoff.md` 中仍保留手机型号、Android 版本、安装启动结果、截图和耗时等待回填项。这个风险不影响仓库中 APK 的存在，但会影响最终测试报告的完整度。

建议收尾时至少补一组真机记录：

| 项目 | 建议记录 |
| --- | --- |
| 手机型号 | 具体品牌和型号 |
| Android 版本 | 系统设置中的 Android 版本 |
| 测试图片 | 图片格式、分辨率、来源 |
| 运行环境 | 是否断网、是否首次安装 |
| 结果 | 是否成功选图、是否显示预览、是否生成直方图、耗时多少 |
| 证据 | 截图或录屏 |

### 6.2 文档一致性风险

项目后期 UI、loading、Histogram Compare、保存图片、launcher 图标等内容更新较多，如果文档只写最早版本，会显得项目能力不足；如果写得过满，又可能超过实际证据。

建议采用下面的写法：

- 核心要求写成“已实现并有测试”。
- UI 与保存类功能写成“答辩展示增强”。
- 真机耗时写成“待回填或补充实测表”。
- 开发机 benchmark 写成“同边界策略对照”，不替代真机结论。

### 6.3 答辩现场风险

答辩时最容易出问题的不是算法本身，而是演示节奏。建议现场只走一条稳定路径：

```text
打开正式演示版本 APK
-> 断网说明
-> 选择准备好的本地图片
-> 展示原图预览
-> 切到输出页看 256x100 直方图
-> 切到指标页解释 bins、像素总数、耗时和 300ms
-> 如有时间，再打开低效压力对照版本 APK 做对比
```

不要现场临时挑超大图片，也不要把保存图片、横屏放大等增强功能放在主演示路径里。

## 8. 项目经理结论

目前风险总体可控。项目最大的技术风险已经通过主流程实现、算法测试、离线检查和性能对照降低了；剩余风险更多是交付管理问题，包括真机证据回填、正式测试报告、使用说明和答辩材料。

五天周期项目最怕第 5 天继续扩功能。当前更稳的做法是停止加新能力，把已有成果讲清楚、证据补完整、演示路径练稳定。
